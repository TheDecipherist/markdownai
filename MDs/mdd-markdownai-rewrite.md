# MDD × MarkdownAI — Complete Rewrite Plan

**Status:** Ready to execute  
**Source analyses:** `.mdd/docs/45-mdd-markdownai-integration.md` + `.mdd/docs/46-mdd-token-optimization-analysis.md`  
**Target:** `~/projects/mdd` — the MDD workflow tool  
**Goal:** Rewrite every MDD mode file and artifact using MarkdownAI so the workflow tool that enforces "document first" itself uses "documentation that cannot lie."

---

## What MDD Is Today

MDD (`~/projects/mdd`) is a collection of markdown instruction files Claude loads lazily per invocation. They live at `~/.claude/mdd/` after global install. Claude reads them as prompt instructions and follows them manually.

```
~/.claude/mdd/
  mdd.md              221 lines — router, mode dispatch, branch guard, worktree
  mdd-build.md        780 lines — Phases 0–7d: understand → document → test → implement → verify
  mdd-audit.md        380 lines — parallel multi-agent code review + manifest resume
  mdd-manage.md       481 lines — status, scan, note, update, deprecate, plan-execute
  mdd-plan.md         471 lines — plan-initiative, plan-wave, plan-execute
  mdd-lifecycle.md    376 lines — reverse-engineer, graph, upgrade
  mdd-ops.md          382 lines — ops runbooks, execution
  mdd-import-spec.md  622 lines — spec→feature docs conversion
  mdd-manual.md       441 lines — user manual generation
  ─────────────────────────────
  Total:            4,154 lines — ~49,846 tokens for the full system
```

Per session (lazy loading: router + one mode file + `.startup.md`):
- **~10,234 tokens** of MDD infrastructure loaded before Claude does any work

The project also produces artifacts in `.mdd/` inside each managed project:
- `.mdd/.startup.md` — 141 lines, static, drifts between sessions
- `.mdd/connections.md` — 183 lines, regenerated by Claude after every doc change
- `.mdd/docs/*.md` — feature docs, static markdown, can lie about status
- `.mdd/audits/*.md` — audit reports, static, go stale as code changes
- `.mdd/ops/*.md` — runbooks, instructions a human follows manually

---

## The Core Problem

All of this is static markdown. It is accurate when written and drifts silently thereafter. The irony: **MDD enforces "document first, keep docs as source of truth" — but its own infrastructure drifts and lies.**

MarkdownAI fixes this. Any `.md` file gains live data, conditional rendering, macro composition, phase-aware workflows, and MCP-backed execution by adding a `@markdownai` header. No new file extension needed.

## What Changes, File by File

### 1. `.mdd/.startup.md` → Live Document

**Problem:** Static. Between `/mdd status` runs it shows wrong branch, wrong feature count, stale audit summary. Claude acts on wrong state.

**Fix:** Add `@markdownai` header. A pre-session hook runs `mai render .mdd/.startup.md` before Claude enters every session.

**What the live version queries:**
```markdown
@markdownai
@date format={YYYY-MM-DD HH:mm}
@query git branch --show-current
@query git log --oneline -5
@query git remote get-url origin 2>/dev/null || echo "(no remote)"
@count .mdd/docs/ "status: complete"
@count .mdd/docs/ "status: in_progress"
@count .mdd/docs/ "status: draft"
@list .mdd/docs/*.md format={numbered}
@read .mdd/audits/report-*.md @sort=mtime @limit=1
```

**Token impact:** Eliminates 750–1,500 tokens of mid-session re-verification queries Claude currently runs to check project state. More importantly: **eliminates the entire class of "stale context" errors** where Claude acts on wrong assumptions.

**Hook to add (MDD install):**
```bash
# ~/.claude/hooks/mdd-pre-session.sh
if [ -f ".mdd/.startup.md" ] && grep -q "@markdownai" .mdd/.startup.md; then
  mai render .mdd/.startup.md --output .mdd/.startup-rendered.md
fi
```

### 2. `.mdd/connections.md` → Live Document

**Problem:** Claude rebuilds this file from scratch after every doc create/modify/archive. The rebuild spec (10 lines) appears in 6 of 8 mode files — copied 6 times. Any drift means the file is wrong.

**Fix:** Add `@markdownai` header. `mai render .mdd/connections.md` replaces Claude's manual rebuild entirely. The rebuild spec disappears from all 6 mode files (60 lines removed system-wide).

**What the live version generates:**
```markdown
@markdownai
@list .mdd/docs/*.md format={json}
@graph depends_on from={.mdd/docs/*.md} style={mermaid}
@tree .mdd/docs/ depth=2
@count .mdd/docs/ "depends_on: \[\]" label="no dependencies"
```

**Token impact:** ~60 lines of repeated spec removed from mode files (~720 tokens), plus zero Claude execution cost for rebuilds.

**CI integration:** `mai validate .mdd/connections.md` catches broken `depends_on` references automatically. Add to pre-commit or CI.

### 3. Audit Reports → Live Documents

**Problem:** Audit reports are static. A finding like "checkFilePath not called at line 45" may have been fixed. The report still shows it as open. Claude re-reads source files to verify — 2–4 reads per finding, ~8,000–16,000 tokens per audit resolution session.

**Fix:** Add `@markdownai` header to audit report template. Each finding embeds the live code it references:

```markdown
@markdownai

### P1 — checkFilePath not called before file read

**Live code at finding location:**
@include packages/engine/src/executeInclude.ts lines=43-55

**Expected:** `checkFilePath(resolvedPath)` called before `fs.readFile`
**Finding date:** 2026-05-16
**Resolution:** Visible above — if the call appears, this finding is closed.
```

When the fix is applied, `mai render` of the audit report shows the current code inline. No re-reads. No guessing. **Finding still open vs finding resolved is visible in one render.**

**Token impact:** ~400–800 tokens saved per finding (eliminates source file re-read). For a 20-finding report: **8,000–16,000 tokens saved** in the resolution session.

### 4. Ops Runbooks → Executable Documents

**Problem:** Ops runbooks are instructions. "Executing" one means Claude reads and follows manually. No self-checking — Claude could be wrong about whether a health check passed.

**Fix:** Add `@markdownai` header. Use `@phase` for pre-flight/deploy/post-flight gates. Use `@http` and `@env` for self-validation.

```markdown
@markdownai

@phase pre-flight
  @env DEPLOY_TOKEN required="true"
  @env IMAGE_NAME required="true"
  @http GET {{ HEALTH_ENDPOINT }}/health expected=200
  @query docker image ls {{ IMAGE_NAME }} | head -2
@on complete
  Pre-flight passed — all checks green.
@end

@phase deploy
  @if @env PRODUCTION
    @query kubectl set image deployment/{{ APP_NAME }} {{ IMAGE_NAME }}:{{ TAG }}
  @else
    @query docker-compose up -d {{ APP_NAME }}
  @endif
@on complete
  Deployment complete.
@end

@phase post-flight
  @http GET {{ HEALTH_ENDPOINT }}/health expected=200 retries=5 delay=10s
  @query kubectl get pods -l app={{ APP_NAME }}
@end
```

`mai render ops/deploy-production.md` executes the runbook. Pre-flight failures stop before any deployment. `@env required` prevents running deploys with missing credentials.

### 5. Mode Files — Shared Library Extract

**Problem:** Six patterns are duplicated across mode files. Every load pays the cost:

| Pattern | Occurrences | Lines wasted |
|---|---|---|
| Branch guard logic | 3 files (mdd-build, mdd-plan ×2, mdd-manage) | ~123 lines |
| `connections.md` regen spec | 6 files | ~60 lines |
| `.startup.md` rebuild reference | 8 files, 19 occurrences | ~57 lines |
| Green Gate loop | 2 files | ~40 lines |
| Commit/merge flow | 2 files | ~20 lines |
| Hash computation spec | 2 files | ~50 lines |
| **Total** | | **~350 lines** |

**Fix:** Extract into `~/.claude/mdd/mdd-shared.md` with `@markdownai` header and `@define` macros. Each mode file `@import`s it and uses `@call`.

```markdown
@markdownai
<!-- mdd-shared.md — imported by all mode files -->

@define branch-guard
  @query git branch --show-current
  @query git status --porcelain
  @if {{ branch }} == "main" or {{ branch }} == "master"
    @if {{ dirty }} != ""
      @constraint STOP — uncommitted changes on {{ branch }}.
      Options: (a) commit (b) stash (c) abort
    @else
      @query git checkout -b feat/{{ slug }}
      ✅ Branched to feat/{{ slug }}
    @endif
  @endif
@end

@define rebuild-connections
  @list .mdd/docs/*.md format={json}
  @graph depends_on from={.mdd/docs/*.md} style={mermaid}
  @tree .mdd/docs/ depth=2
@end

@define rebuild-startup
  @query mai render .mdd/.startup.md
@end

@define green-gate
  Run: pnpm test:unit -- --grep "{{ feature }}" && pnpm typecheck
  Max iterations: 5
  @constraint Never modify tests. If test seems wrong, re-read MDD doc first.
  @constraint If doc seems wrong, STOP and ask the user before changing anything.
  @constraint Iteration 5 exhausted → STOP, report options to user.
@end

@define commit-merge-flow
  1. Stage: git add -A
  2. Commit: use /commit skill
  3. Merge: git checkout main && git merge feat/{{ feature }} --no-ff
  4. Ask: "Push to origin? (yes/no)"
@end
```

Each mode file then becomes:
```markdown
@markdownai
@import ~/.claude/mdd/mdd-shared.md

## Step 0 — Branch Guard
@call branch-guard
```

**Token impact per session:** ~70 lines eliminated (branch guard alone: 63→2 lines in mdd-build.md). **~840 tokens/session.**

### 6. Mode Files — Conditional Sections

**Problem:** Sections load into Claude's context even when Claude immediately skips them. In mdd-build.md alone:

| Section | Skip condition | Lines loaded pointlessly |
|---|---|---|
| Phase 2 — Data Flow Analysis | Greenfield (<5 source files, no existing docs) | 65 lines |
| Phase 7b — Integration type blocks | Only 1 of 4 types applies | ~50 lines avg |
| Phase 0 — Branch mismatch check | Branch name matches feature | ~8 lines |

**Fix:** Wrap with `@if` that checks live state:

```markdown
@markdownai
@count .mdd/docs/ label=doc_count
@count src/**/*.ts label=ts_count

@if {{ doc_count }} > 0 and {{ ts_count }} > 4
### Phase 2 — Data Flow Analysis
...65 lines only rendered when not greenfield...
@endif
```

For integration type gates:
```markdown
@read .mdd/docs/{{ active_doc }}.md format={json} path={$.routes} label=has_routes
@list src/components/ label=has_components

@if {{ has_routes }} != ""
  ### Backend verification...
@endif
@if {{ has_components }} != ""
  ### Frontend verification...
@endif
```

**Token impact:** ~400–780 tokens/session depending on project type. 100% of greenfield projects eliminate Phase 2 (65 lines = 780 tokens).

### 7. Mode Files — `@ai-format` Narrative Stripping (Wave 5)

**Problem:** 37% of mode file content is narrative prose — explanations of *why*, context for human readers. Claude trained on this material doesn't need the why re-explained each session.

mdd-build.md example of prose that adds no machine value:
> "Before gathering any context, verify the current branch is compatible with the requested feature. The goal here is to prevent accidental mixing of unrelated features on the same branch, which makes PR review, rollback, and history harder — and breaks the MDD workflow."

With `@consumer=ai` mode (Wave 5):
```markdown
@markdownai
@ai-format strip=narrative

@constraint Branch must match feature slug. Mismatch → prompt user.
```

**Token impact per session:** ~1,416 tokens eliminated (118 lines of narrative stripped per average mode file at 65% compression of the 37% narrative proportion). For mdd-build.md specifically: **~2,736 tokens** (228 lines from 350 narrative lines).

**This is Wave 5** — implement after Wave 5 (`34-ai-consumer-mode` through `39-ai-format`) is complete.

### 8. Mode Files — `@phase` + MCP Phase Navigation (Wave 4 MCP)

**Problem:** mdd-build.md loads all 780 lines. Claude is in Phase 3 but has Phases 4–7 (test skeletons, build plan, implementation, verification — ~480 lines) fully in context doing nothing.

**Fix:** Restructure mode files using `@phase` blocks. MCP server tracks current phase. Claude only has the active phase in context.

```markdown
@markdownai

@phase understand title="Phase 1 — Understand the Feature"
  Launch 3 parallel Explore agents...
  Ask user questions...
@on complete
  next_phase: document
@end

@phase document title="Phase 3 — Write MDD Documentation"
  Create .mdd/docs/NN-feature-name.md...
@on complete
  next_phase: test-skeleton
@end

@phase test-skeleton title="Phase 4 — Generate Test Skeletons"
  ...
@on complete
  next_phase: red-gate
@end
```

Claude calls `list_phases` (MCP) → knows current phase. `next_phase` → advances. Phase state persists across context compaction. After a compaction event, `list_phases` re-orients Claude in one call.

**Token impact:** ~660 lines hidden at any moment = **~7,920 tokens** for a mid-session Phase 3 call. Entire Phases 1–2 and 4–7 are not in context while Claude is working in Phase 3.

**Accuracy impact:** Phase-specific rules are at the *top* of context when relevant, not buried 500 lines back. Compliance with specific phase gates improves **20–35%**.

**This requires MCP E2E validation** (docs 41–44) to be complete first.

### 9. Feature Docs — Live Health Section

**Problem:** Feature docs can say `status: complete` while source files are missing. Caught by audit only, not on casual inspection.

**Fix:** Add `@markdownai` header to feature docs. Live health section renders on every `mai render`:

```markdown
@markdownai

---
status: complete
source_files:
  - packages/engine/src/executeInclude.ts
  - packages/engine/src/executeQuery.ts
---

## Live Health

@if @list packages/engine/src/executeInclude.ts returns empty
  ⚠️ MISSING: packages/engine/src/executeInclude.ts
@endif
@if @list packages/engine/src/executeQuery.ts returns empty
  ⚠️ MISSING: packages/engine/src/executeQuery.ts
@endif
@query pnpm typecheck --filter=@markdownai/engine 2>&1 | tail -3
```

`mai render .mdd/docs/03-engine.md` instantly shows whether the feature is actually complete. No audit run needed.

## New Fields for MDD Feature Docs

When MDD manages a project that produces live markdown documents, the feature doc needs these fields:

```yaml
live_documents:
  - .mdd/.startup.md
  - .mdd/connections.md
directives_used: [read, list, query, graph, phase, define, if, count]
```

MDD audit should run `mai validate` on all `live_documents` as part of Phase 7c. The `/mdd validate-live` mode (new) runs `mai validate` on all `@markdownai` documents in the project.

## The Philosophical Point (Why This Matters)

MDD's core claim is "document first, document as source of truth." Every project it manages gets that guarantee enforced by the workflow. But MDD's own infrastructure — `.startup.md`, `connections.md`, audit reports, ops runbooks — is static markdown that drifts.

The rewrite closes this loop. When MDD itself uses MarkdownAI:

- **`.startup.md` cannot lie about feature count or branch** — it renders from live data
- **`connections.md` cannot reference a doc that doesn't exist** — `@graph` fails validation if the dep is missing
- **Audit reports cannot pretend a finding is open after it's fixed** — `@include` shows current code
- **Ops runbooks cannot skip a health check** — `@http` either passes or halts the phase

MDD becomes a workflow tool whose own documentation infrastructure embodies the property it enforces in every project it manages: **documentation that cannot lie.**

## Quick Reference: MarkdownAI Features Used and Where

| Feature | Where used in rewritten MDD |
|---|---|
| `@markdownai` header | All mode files, all artifact templates |
| `@define` / `@call` | Shared library: branch guard, green gate, connections rebuild |
| `@import` | Every mode file imports mdd-shared.md |
| `@if` / `@endif` | Skip-conditional sections, integration type gates, env-gated runbook steps |
| `@query` | Live git state, test runs, health checks |
| `@count` | Feature doc counts in startup, health checks |
| `@list` | Source file existence checks, doc enumeration |
| `@read` | Active doc frontmatter, package.json fields |
| `@include` | Audit findings embed live source code with line ranges |
| `@date` | Startup timestamp, last-rendered tracking |
| `@env` | Required credential checks in runbooks |
| `@http` | Health checks in ops runbook phases |
| `@graph` | Connections map dependency visualization |
| `@tree` | Connections map path tree |
| `@phase` / `@on complete` | Mode file workflow phases (Wave 4 MCP) |
| `@constraint` | Machine-readable rules replacing prose |
| `@ai-format` | Narrative stripping for AI-consumption (Wave 5) |
| `@consumer=ai` | AI-format rendering of mode files (Wave 5) |
