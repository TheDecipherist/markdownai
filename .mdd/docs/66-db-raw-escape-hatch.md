# 66 — DB — raw= Escape Hatch

## What to Build

This SPEC describes the `raw=` operation on the @db directive: how it works, what security it requires, what audit trail it always generates, and why it is never cached automatically. The implementing COMPONENT is `68-db-executor` via the `executeRaw()` method on each adapter.

The `raw=` option accepts a native query string and passes it directly to the database adapter, bypassing the QueryPlan translation layer entirely. It is the escape hatch for queries that cannot be expressed in the structured query language: complex joins, CTEs, window functions, full aggregation pipelines.

## Architecture

Raw queries bypass the QueryPlan entirely. The executor checks `allow_raw` in the security config and either passes the raw string to `adapter.executeRaw(query)` or strips the directive with a WARN. Raw query strings never travel through the QueryPlan type system.

The immutable block pattern list (`72-db-security`) is applied to raw queries at the directive level, before the executor is called - even with `allow_raw: true`.

## Implementation Notes

`raw=` must be classified as a distinct operation by the parser - not as a value parsed into a QueryPlan. When `raw=` is present, the executor routes directly to `executeRaw()` on the adapter.

The WARN generated by `raw=` execution must be unconditional - it fires regardless of `--silent`, log level, or security mode. This is the explicit design choice: raw queries bypass the safety guarantees of the translation layer and deserve explicit logging.

## Data Model

**Security config for raw= (per connection):**

```json
{
  "db": {
    "primary": {
      "allow_raw": true
    }
  }
}
```

`allow_raw` defaults to `false`. It must be explicitly set to `true` per named connection.

## API / Interface

**SQL raw query:**

```markdown
@db using="primary" raw="SELECT u.name, COUNT(o.id) as orders FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE u.active = true GROUP BY u.id ORDER BY orders DESC LIMIT 10" | @render type="table"
```

**MongoDB raw aggregation pipeline:**

```markdown
@db using="analytics" raw='db.events.aggregate([{"$group":{"_id":"$type","count":{"$sum":1}}},{"$sort":{"count":-1}}])' | @render type="table"
```

## Business Rules

1. `raw=` requires `allow_raw: true` in `~/.markdownai/security.json` for the named connection. Without it, the directive is stripped with a WARN and a clear message explaining how to enable it.
2. `raw=` always generates a WARN in the audit log when it executes, regardless of security mode, `--silent`, or any other flag. This behavior is intentional and cannot be suppressed.
3. `raw=` is never cached automatically. The author must add `@cache` explicitly if caching is needed.
4. `raw=` does not produce a QueryPlan. The raw string is passed directly to `adapter.executeRaw()`.
5. The immutable block patterns from `72-db-security` are applied to raw queries before execution, even with `allow_raw: true`. Raw queries matching a block pattern are rejected with a SECURITY_ALERT, not a WARN.
6. `raw=` cannot be combined with any other operation option on the same directive. It is mutually exclusive with `find=`, `one=`, `count=`, and `aggregate=`.
7. `raw=` is compatible with `using=` and `uri=` for connection selection, and with `@cache` for explicit caching.

## Acceptance Criteria

- @db with `raw=` and no `allow_raw: true` in security config is stripped with a WARN (not an error, not silent)
- @db with `raw=` and `allow_raw: true` executes the raw string via `adapter.executeRaw()`
- Every `raw=` execution writes a WARN to the audit log even with `--silent` or minimal log level
- `raw=` combined with `find=` on the same directive raises a FATAL parse error
- A raw SQL query containing `DROP TABLE users` is blocked by immutable rules even with `allow_raw: true`, raising SECURITY_ALERT
- A raw MongoDB query containing `db.users.drop()` is blocked by immutable rules even with `allow_raw: true`
- `raw=` without explicit `@cache` produces a fresh query result every render (no automatic caching)
- `raw=` with `@cache session` caches and replays correctly
